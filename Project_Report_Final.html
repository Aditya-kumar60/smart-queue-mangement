1. Abstract

In modern healthcare facilities, patient queue management remains a significant logistical hurdle. Traditional physical queuing systems often result in overcrowding in waiting rooms, increased patient anxiety, and inefficient utilization of administrative staff time. This project proposes and implements the "Smart Queue & Appointment Management System," a robust web-based application designed to digitize, streamline, and automate the patient queuing and appointment tracking process.

By leveraging real-time web technologies, the system completely removes the need for physical lines. It assigns digital tokens to patients and broadcasts live status updates across all connected devices instantly without page reloads. The solution is strictly role-based, incorporating discrete operational interfaces for Patients, Doctors, and administrative Staff, ensuring data privacy and operational efficiency.

2. Introduction

The healthcare industry relies heavily on operational efficiency to provide timely medical care. However, the patient intake process remains largely manual in many clinics and mid-sized hospitals. When patients arrive without foresight into doctor availability or active queue lengths, bottlenecks inevitably form at the reception desk.

The Smart Queue Management System was developed to bridge this gap. Rather than waiting in a potentially infectious waiting room for hours, patients can secure a digital token remotely or via a walk-in registration handled by staff. The system introduces true multi-tenant queue handling whereby every individual doctor has an isolated, sequentially numbered queue starting from token number one. This prevents systemic token overlap and ensures absolute clarity for both the medical provider and the patient.

3. Problem Statement

Currently, the management of outpatient departments (OPDs) and private clinics faces several critical challenges:

  • Physical Crowding: Lack of digital tracking forces patients to remain physically present to secure their position in line, leading to uncomfortable and unhygienic waiting areas.
  • Opaque Wait Times: Patients have zero visibility into how many individuals are ahead of them or the approximate time until their consultation.
  • Clerical Overload: Reception desk staff spend a disproportionate amount of time simply answering "what is the current token number?" rather than focusing on medical admittance.
  • Scattered Medical Records: Diagnosis and prescription data is often mismanaged across disparate paper files.

4. Objectives of the Project

The primary aim of this system is to entirely abstract the physical waiting room into a digital interface. Key objectives include:

  • To design an intuitive token generation mechanism that is unique to each respective medical practitioner.
  • To implement a seamless real-time broadcasting system so that patients can track the live incrementing of tokens on their personal devices.
  • To provide secure authentication gateways, including Google OAuth 2.0, ensuring that patient data is protected.
  • To establish a secure vault for historical medical records, attaching diagnoses and prescriptions directly to completed token appointments.
  • To empower hospital staff with analytical dashboards to monitor overarching clinic metrics in real-time.

5. Functional & Non-Functional Requirements

5.1 Functional Requirements

  • User Authentication: System must allow patients, doctors, and staff to log in securely using credentials or Google OAuth.
  • Appointment Booking: Patients must be able to view available doctors and book an appointment, receiving a specific queue token.
  • Queue Management: Doctors must be able to view their specific queue, call the next patient, and mark consultations as complete.
  • Walk-in Handling: Staff must be able to manually generate tokens and assign them to doctors for walk-in patients.
  • Medical Records: Once a consultation is complete, the doctor must be able to input the diagnosis and prescription, which is saved to the patient's history.

5.2 Non-Functional Requirements

  • Real-Time Updates: Queue changes must be broadcasted instantly to all connected clients without requiring a manual page refresh (Latency < 1 second).
  • Security: All user passwords must be hashed using bcrypt. Access to role-specific dashboards must be protected via JWT authorization.
  • Reliability & Availability: The system must handle multiple concurrent booking requests without overlapping token assignment.
  • Responsiveness: The user interface must be fully responsive and operational on both mobile devices and desktop computers.

6. Technologies Used

The project was conceptualized and developed using a modern, JavaScript-centric Full-Stack (MERN-inspired) architecture. This ensures high performance, non-blocking I/O operations, and rapid data transmission.

6.1 Frontend Technologies

  • HTML5 & CSS3: Used to construct the structural foundation and aesthetic layer of the application. The dashboards utilize modern CSS Flexbox and CSS Grid designs to ensure responsive behavior across mobile and desktop environments.
  • Vanilla JavaScript (ES6+): Chosen over heavier frameworks (like React or Angular) to maintain a lightweight footprint. It efficiently manages DOM manipulations, asynchronous API calls via the Fetch API, and client-side form validations.

6.2 Backend Technologies

  • Node.js: Serves as the runtime environment. Its asynchronous, event-driven nature naturally complements the heavy I/O operations required by a real-time queuing system.
  • Express.js: A minimal and flexible Node.js web application framework used to establish robust RESTful API endpoints and middleware logic for authentication and routing.
  • Socket.io: A critical library enabling bidirectional, event-based communication between web clients and servers. It is the engine behind the "Live Queue" feature, instantly pushing token updates to patient dashboards the moment a doctor progresses their queue.

6.3 Database & Security

  • MongoDB: A NoSQL document database used to store unstructured/semi-structured data like user profiles, nested appointment histories, and consultation notes.
  • Mongoose: An Object Data Modeling (ODM) library for MongoDB that enforces data schemas, relationships (e.g., linking a Medical Record strictly to a specific User ID), and validation rules at the application layer.
  • JWT (JSON Web Tokens) & Bcrypt: Employed to facilitate stateless, highly secure user authentication sessions and robust password hashing.
  • Google OAuth 2.0: Integrated via the Google Identity Services API, providing users with a frictionless "Sign-in with Google" capability, improving onboarding speed while maintaining enterprise-grade security.

7. System Architecture

7.1 Architecture Overview

The system follows a three-tier architecture consisting of the Presentation Layer (Frontend), Business Logic Layer (Backend), and Data Layer (Database). The frontend communicates with the backend through REST APIs, and real-time communication is handled through Socket.io.

Layer Components
Presentation Layer HTML, CSS, JavaScript, Dashboard Pages
Business Logic Layer Node.js, Express.js, Controllers, Routes, Middleware
Data Layer MongoDB, Mongoose Models (User, Appointment, Medical)
Real-time Layer Socket.io for live queue updates

7.2 MVC Pattern

The backend follows the MVC (Model-View-Controller) architectural pattern:

  • Model: MongoDB schemas defined in Mongoose (User.js, Appointment.js, Medical.js)
  • View: HTML pages served as static files from the Frontend folder
  • Controller: Business logic functions in authController.js, patientController.js, doctorController.js, staffController.js

8.3 Complete System Flow

The following steps describe the complete workflow of the system:

  1. Doctor registers on the portal with role as 'doctor'. Data is saved in MongoDB.
  2. Patient registers and logs into the system with role as 'patient'.
  3. Patient selects a doctor from the dropdown and books an appointment.
  4. System generates a sequential token number specific to that doctor's queue and saves the appointment in MongoDB.
  5. Patient dashboard shows the assigned token number and current running token.
  6. Doctor logs in and sees all waiting patients in the queue sorted by token number.
  7. Doctor clicks 'Next Patient' button. The current patient status changes to in-progress.
  8. Socket.io emits a real-time event. All connected patient dashboards automatically update the current token without page refresh.
  9. Doctor enters diagnosis and prescription and clicks 'Complete Consultation'.
  10. Appointment is marked completed. Medical record is saved. Patient's medical history updates in real time.

9. Core Modules & Functionality

The system is divided into four main modules based on functionality:

9.1 Authentication Module

This module handles user registration and login for all three roles. It uses JWT (JSON Web Token) for secure authentication and bcryptjs for password hashing.

  • User Registration: Collects name, email, password, and role. Validates inputs, checks for duplicate email, hashes the password, and saves to MongoDB.
  • User Login: Verifies email and password, generates a JWT token, and redirects the user to their role-based dashboard.
  • Role-Based Redirect: After login, patients go to patient.html, doctors to doctor.html, and staff to staff.html.
  • JWT Protection: All dashboard API calls require a valid JWT token in the Authorization header.

9.2 Patient Module

  • Authentication & Profile: Secure login via standard credentials or Google OAuth.
  • Doctor Selection: Dynamic fetching of currently available medical practitioners.
  • Token Generation: Upon confirming an appointment, the backend calculates the highest current token for that specific doctor and assigns `$Token + 1` to the patient.
  • Live Tracking: Patients view an active dashboard listening to WebSockets. If the doctor updates the queue, the patient sees the "Current Token Being Served" change instantly.
  • Medical Vault: A read-only view of historical diagnoses and prescriptions linked to their profile.

9.3 Doctor Module

  • Queue Management: Doctors see a sorted list of their specific upcoming patients. They have granular control via a "Call Next Patient" trigger mechanism.
  • State Encapsulation: By pressing "Next", the system automatically finds the active patient, transitions their database state to `completed`, and promotes the next waiting patient to `in-progress`.
  • Consultation Finalization: The interface provides text areas for inputting clinical diagnoses and prescriptions, permanently writing them to the MongoDB `Medical` collection.

9.4 Staff/Admin Module

  • Walk-in Management: Recognizes that not all hospital demographics are digitally literate. Staff can manually register walk-in patients, assigning them standard digital queue tokens that seamlessly merge with online bookings.
  • Queue Auditing: Staff can view macroscopic data, canceling rogue or duplicate appointments globally to maintain queue integrity.
  • Clinic Analytics: High-level computational endpoints return daily throughputs, total waiting counts, and active doctor availability numbers.

10. Real-Time Concurrency Execution

A flagship technical achievement of this project is the integration of Socket.io to solve the classic "polling problem".

In standard web applications, a client must repeatedly ask the server (Long-Polling) if the queue token has changed, heavily degrading server CPU and network bandwidth. In this project, we implement full-duplex TCP persistent connections. When a Doctor clicks "Next Patient", the Express backend executes the database write, and concurrently triggers `io.emit('queueUpdated')`. Every patient's browser, holding an open socket, receives this payload asynchronously and updates the DOM in milliseconds. This results in incredibly low latency and high scalability.

11. Conclusion

The Smart Queue & Appointment Management System successfully demonstrates how classical operational bottlenecks in localized healthcare can be entirely neutralized through strategic software engineering. By isolating queues per medical practitioner and broadcasting state changes through web sockets, the system achieves a highly transparent, fair, and organized environment.

The utilization of a lightweight Node/Express stack ensures that the application remains extremely responsive, while the separation of interfaces ensures data privacy and limits the operational complexity for end-users. The project stands as a fully viable Minimal Viable Product (MVP) ready for small-to-medium institutional deployment.
