1. Introduction

1.1 Project Overview

The Smart Queue and Appointment Management System is a comprehensive web-based application designed to digitize hospital queue management. It replaces traditional physical queues with an efficient, technology-driven digital system. Patients can seamlessly book appointments, monitor their queue positions in real-time, and receive timely notifications throughout their healthcare journey.

The system supports three distinct user roles: Patients, Doctors, and Staff (Receptionists/Administrative Personnel). Each role has a tailored interface and workflow, ensuring a streamlined experience for all stakeholders in the healthcare facility. Real-time updates powered by WebSocket technology enable instant synchronization across all connected users.

1.2 Problem Statement

Traditional hospital queue systems face several challenges:

  • Long waiting times leading to patient dissatisfaction
  • Manual token management prone to human errors
  • Lack of real-time visibility for patients and doctors
  • Inefficient staff coordination and appointment management
  • No automated notifications or reminders

1.3 Project Objectives

  • Replace physical queues with a completely digital system
  • Enable patients to book appointments online with automatic token generation
  • Provide real-time queue tracking and position monitoring
  • Facilitate doctor-patient consultations with medical record documentation
  • Empower staff with tools to manage walk-ins and administrative tasks
  • Ensure secure, role-based access with JWT authentication
  • Implement real-time updates using Socket.io for instant feedback
  • Send automated email notifications for key system events

1.4 Project Scope

This project encompasses:

  • User registration and JWT-based authentication
  • Appointment booking with automatic token generation
  • Real-time queue management and status tracking
  • Doctor consultation interface with medical note recording
  • Staff dashboard for administrative operations
  • Email notification system for users
  • Responsive web interface for desktop browsers
  • MongoDB database with Mongoose ODM

1.5 Development Methodology

5.3 Key Implementation Features

5.3.1 JWT Authentication Middleware:

Protects all private routes by verifying JWT tokens from request headers. Ensures only authenticated users access sensitive APIs. Token contains user ID and role for authorization decisions.

5.3.2 Real-Time Queue Updates:

When a doctor calls "Next Patient", the server emits a socket event broadcasted to all connected clients. Patients instantly see queue position updates without page refresh. Implemented via io.emit('queueUpdated').

5.3.3 Token Generation Logic:

Automatic token numbers assigned based on doctor's current queue. Prevents duplicates through database queries before assignment. Numbers increment sequentially per doctor.

5.3.4 Email Notifications:

Nodemailer sends automated emails for:

  • Appointment confirmation to patient and doctor
  • Token called notification when patient's turn arrives
  • Consultation completion with diagnosis/prescription

6. Features and Functionality

6.1 User Registration and Authentication

Users can register with name, email, password, and role selection. Passwords are hashed with bcryptjs (10-round salting). Login generates a JWT token valid for 24 hours. Token stored in localStorage on frontend for subsequent API calls.

  • Email uniqueness validation
  • Password strength validation
  • Role-specific redirects after login
  • Secure logout clearing local storage

6.2 Patient Features

View Doctors List:

Fetches all doctors from database using GET /api/patient/doctors. Displays in dropdown for appointment selection.

Book Appointment:

POST /api/patient/appointments creates new appointment. System prevents duplicate active appointments with same doctor. Token auto-generated based on queue status.

Track Queue Position:

  • Displays current token being served
  • Shows waiting count
  • Updates in real-time via Socket.io
  • Shows appointment status (waiting/in-progress/completed)

View Medical History:

GET /api/patient/medical-records retrieves past consultations with diagnosis and prescription details.

6.3 Doctor Features

View Queue:

GET /api/doctor/queue shows all waiting and in-progress appointments. Displays current token and waiting count. Real-time updates via Socket.io.

Call Next Patient:

POST /api/doctor/next marks current patient as completed and brings next waiting patient to "in-progress". Sends email notification to patient.

Complete Consultation:

POST /api/doctor/complete records diagnosis and prescription in Medical collection. Updates appointment status to "completed". Sends completion email to patient.

6.4 Staff Features

Add Walk-in Patient:

POST /api/staff/walkin adds patient without prior appointment. Assigns token and stores in queue.

Manage Appointments:

  • View all appointments with filters
  • Cancel appointments with DELETE /api/staff/appointments/:id
  • Update appointment status

Statistics Dashboard:

  • Total appointments today
  • Patients waiting
  • Available doctors
  • Completed consultations

6.5 Real-Time Features

All clients listen to Socket.io events. When queue changes, server broadcasts 'queueUpdated' event. Clients reload relevant data without manual refresh. Implements watching pattern for reactive updates.

8. Challenges and Solutions

8.1 Real-Time Synchronization

Challenge: Keeping all connected clients updated when queue status changes. Initial implementation broadcast to all users, causing unnecessary data transfers.

Solution: Implemented Socket.io rooms and targeted broadcasts. Each doctor's queue broadcast only affects relevant patients and staff. Reduces network overhead significantly.

8.2 Concurrent Appointment Bookings

Challenge: Multiple patients booking with same doctor simultaneously could generate duplicate token numbers.

Solution: Added database transaction and pre-booking validation. Queries existing appointments before token assignment. Prevents race conditions.

8.3 Database Schema Flexibility

Challenge: Requirement to potentially add features like appointment duration or doctor specialities meant schema needed flexibility.

Solution: Used MongoDB with Mongoose validation allowing optional fields. Schema can evolve without breaking existing data.

8.4 Email Service Reliability

Challenge: Email sending could fail, potentially blocking API responses.

Solution: Implemented non-blocking email sending. Email sends asynchronously without awaiting completion. API responds immediately while email processes in background.

8.5 Cross-Origin Requests

Challenge: Frontend and backend run on different ports (5000), causing CORS errors.

Solution: Configured CORS middleware in Express to accept requests from frontend origin. Socket.io CORS also configured.

8.6 Frontend State Management

Challenge: Keeping frontend UI in sync with server state, especially with real-time updates.

Solution: Implemented smart reload functions triggered by Socket.io events. Frontend fetches fresh data from server when updates occur, ensuring consistency.

8.7 Session Management

Challenge: Managing user sessions across browser refreshes.

Solution: Store JWT token in localStorage. Frontend automatically includes token in subsequent API calls until logout. No server-side session storage needed.
